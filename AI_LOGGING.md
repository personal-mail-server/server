# AI_LOGGING.md

## 문서 목적
이 문서는 AI가 수행한 실제 변경 작업의 로그 작성 규칙을 정의한다.

## 저장 위치
- 작업 로그는 `project/.ai/log` 이하에 저장한다.

## 로그 작성 대상
다음과 같이 실제 변경이 발생한 경우 작업 로그를 남긴다.

- 코드 수정
- 문서 수정
- 설정 파일 수정
- CI/CD 수정
- 디렉토리 구조 변경
- 의존성 변경

## 로그 작성 제외 대상
다음과 같은 경우에는 작업 로그를 남기지 않는다.

- 단순 검색
- 정보 조사
- 코드 또는 문서 검토만 수행한 경우
- 사용자 지시에 대한 설명만 수행한 경우
- 실제 파일 변경이 없는 경우

## 작성 원칙
- 로그는 AI가 나중에 검색하고 추적하기 쉬운 고정된 형식으로 작성한다.
- 로그는 실제 변경 작업에 대해서만 남긴다.
- 사람이 읽을 수 있어야 하며, 동시에 AI가 특정 기능, 파일, 변경 시점, 변경 이유를 쉽게 찾을 수 있어야 한다.
- 기능명, 변경 유형, 대상 파일, 관련 키워드는 반드시 명시한다.
- 같은 작업 흐름의 변경은 가능한 한 하나의 로그로 정리한다.

## 파일명 규칙
- 로그 파일은 날짜와 작업 식별자가 드러나는 형식으로 작성한다.
- 권장 형식은 `YYYY-MM-DD_작업요약.md` 로 한다.

예시:
- `2026-03-14_mail-address-grouping.md`
- `2026-03-14_swagger-setup.md`
- `2026-03-15_github-actions-ci.md`

## 로그 템플릿
모든 작업 로그는 아래 형식을 따른다.

```md
# 작업 로그

## 작업 일시
- 2026-03-14

## 작업 유형
- 기능 추가

## 기능명
- 메일 주소 그룹 분류 기능

## 관련 키워드
- 메일
- 그룹
- 분류
- 회사용 메일
- 개인용 메일
- 게임용 메일

## 변경 이유
- 하나의 계정에서 여러 메일 주소를 용도별로 구분해 관리할 수 있어야 하기 때문

## 변경 대상
- `project/code/...`
- `project/code/...`

## 변경 내용
- 메일 주소를 그룹 단위로 관리할 수 있는 구조 추가
- 그룹 조회 및 분류 로직 추가
- 관련 문서 업데이트

## 영향 범위
- 메일 주소 관리 기능
- 그룹 분류 관련 화면 또는 API

## 인덱스 관리
- AI는 작업 로그의 검색성과 추적성을 높이기 위해 필요하다고 판단할 경우 `project/.ai/log/INDEX.md`를 생성하거나 갱신할 수 있다.
- `INDEX.md`의 생성 여부, 갱신 시점, 포함 내용, 정리 방식은 AI가 판단하여 결정한다.
- 단, `INDEX.md`는 작업 로그를 찾기 쉽게 하기 위한 용도로만 사용한다.
- `INDEX.md`에는 로그 파일 목록, 기능명, 작업 일시, 주요 키워드, 관련 로그 경로 등의 검색용 정보를 포함할 수 있다.

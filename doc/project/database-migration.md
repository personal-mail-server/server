# 데이터베이스 마이그레이션 및 롤백 규칙

## 문서 목적
이 문서는 현재 저장소에서 데이터베이스 스키마를 어떻게 변경하고 롤백하는지, 그리고 동기화 작업 중 코드 작성 시 어떤 규칙을 지켜야 하는지를 정의한다.

본 문서는 DB 변경 작업의 기준 문서이며, 개별 마이그레이션 작성 시 이 문서를 우선 참조해야 한다.

---

## 기준 문서
- 현재 구성 문서: `doc/project/current-stack.md`
- 현재 테스트 문서: `doc/project/current-testing.md`
- 문서 저장소 기준: `doc/README.md`
- 현재 구현 기준 파일:
  - `code/internal/db/migrate.go`
  - `code/cmd/migrate/main.go`
  - `code/migrations/`
  - `Makefile`

---

## 현재 구조 요약
- 데이터베이스는 PostgreSQL을 사용한다.
- 업 마이그레이션은 서버 시작 시 자동 적용될 수 있다.
- 롤백은 수동 명령으로만 수행한다.
- 마이그레이션 적용 이력은 `schema_migrations` 테이블로 관리한다.
- 현재 저장소는 외부 ORM 마이그레이션 도구가 아니라 커스텀 SQL 마이그레이션 러너를 사용한다.

---

## 파일 규칙

### 업 마이그레이션
- 위치: `code/migrations/`
- 파일 형식: `NNN_description.sql`
- 예시:
  - `001_create_users.sql`
  - `005_create_refresh_tokens.sql`

### 다운 마이그레이션
- 위치: `code/migrations/`
- 파일 형식: `NNN_description.down.sql`
- 예시:
  - `001_create_users.down.sql`
  - `005_create_refresh_tokens.down.sql`

### 네이밍 규칙
- `NNN`은 0으로 채운 증가형 번호를 사용한다.
- 업 파일과 다운 파일은 같은 베이스 이름을 공유해야 한다.
- 하나의 업 마이그레이션에는 정확히 하나의 다운 마이그레이션이 대응해야 한다.
- 업 마이그레이션 러너는 `NNN_description.sql` 패턴을 만족하는 파일만 업 마이그레이션으로 인식한다.
- 롤백 순서는 최근 적용된 버전부터 역순으로 수행한다.

---

## 실행 규칙

### 업 적용
- 서버 시작 시 `code/internal/app/server.go`를 통해 업 마이그레이션을 자동 적용한다.
- 수동 적용이 필요하면 `make migrate-up`을 사용한다.
- `make migrate-up`은 Compose DB를 기동하고 백엔드 이미지를 통해 마이그레이션 바이너리를 실행한다.

### 롤백
- 롤백은 `make migrate-down STEPS=1`처럼 수동 명령으로만 수행한다.
- 롤백은 최근 적용된 마이그레이션부터 지정한 수만큼만 수행한다.
- 서버 시작 시 자동 롤백은 허용하지 않는다.
- `make migrate-down`도 Compose 네트워크 안에서 전용 마이그레이션 바이너리를 실행하는 것을 기준으로 한다.

### 운영 안전 규칙
- 프로덕션 또는 공유 환경에서 롤백 전에는 영향 범위를 먼저 확인해야 한다.
- 데이터 손실 가능성이 있는 다운 마이그레이션은 특히 신중하게 작성해야 한다.
- 되돌릴 수 없는 변경이라면 다운 파일에서 의도를 명확히 드러내야 하며, 가능한 경우 후속 보정 마이그레이션 방식이 우선이다.
- 과거 데이터 수정 마이그레이션처럼 안전한 역변환을 증명할 수 없는 경우, 다운 파일은 실패를 통해 수동 개입을 요구할 수 있다.

## 코드 작성 규칙

### 1. 이미 적용된 업 마이그레이션은 수정하지 않는다.
- 한 번 공유되었거나 적용된 업 마이그레이션 파일은 내용 변경 대상이 아니다.
- 잘못된 스키마 변경은 기존 파일 수정이 아니라 새 보정 마이그레이션으로 해결한다.

### 2. 스키마 변경은 업/다운을 함께 작성한다.
- 새 테이블, 컬럼, 인덱스, 제약조건 변경 시 업 파일과 다운 파일을 같은 작업 흐름 안에서 같이 작성한다.
- 업만 만들고 다운을 나중으로 미루지 않는다.

### 3. 한 마이그레이션은 하나의 의도를 가져야 한다.
- 여러 독립 변경을 한 파일에 과도하게 묶지 않는다.
- 파일명만 보고도 의도를 추적할 수 있어야 한다.

### 4. 다운 마이그레이션은 실제 역변환 기준으로 작성한다.
- 단순 `DROP`만 가능한 경우에도 실제 되돌림 결과를 의식해 작성한다.
- 데이터 보존이 필요한 변경은 무조건 삭제형 롤백으로 처리하지 않는다.
- 안전한 역변환을 만들 수 없으면 조용한 no-op 대신 명시적 실패로 위험한 롤백을 차단한다.

### 5. 문서 동기화를 함께 수행한다.
- DB 변경 규칙이 바뀌면 `doc/project/current-stack.md`, `doc/project/current-testing.md`, 관련 기준 문서, `.ai/context` 문서를 함께 갱신한다.
- 실제 파일 변경이 있으면 `.ai/log`에 작업 로그를 남긴다.

### 6. 검증 없이 완료 처리하지 않는다.
- 최소 `go test ./...`, `go build ./...`, `go vet ./...` 검증을 통과해야 한다.
- 마이그레이션 구조를 바꿨다면 `make migrate-up`, `make migrate-down STEPS=1` 같은 실행 검증도 함께 확인한다.

---

## 권장 작업 순서
1. 기준 문서와 현재 스키마 상태를 함께 확인한다.
2. 새 업/다운 마이그레이션 쌍을 작성한다.
3. 관련 코드가 새 스키마를 전제로 하는지 확인한다.
4. 테스트와 실행 검증을 수행한다.
5. 현재 구성 문서, 테스트 문서, AI 컨텍스트, 작업 로그를 동기화한다.

---

## 정리
현재 저장소의 데이터베이스 변경은 커스텀 SQL 업/다운 마이그레이션을 기준으로 관리하며, 롤백은 수동 실행만 허용한다.

또한 동기화 작업 중 DB 관련 코드를 작성할 때는 업/다운 쌍 작성, 기존 업 마이그레이션 불변 원칙, 문서·테스트·로그 동기화를 함께 지켜야 한다.

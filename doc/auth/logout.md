# 로그아웃 기능 설계

## 문서 목적
이 문서는 Personal Mail Server의 로그아웃 기능에 대한 요구사항과 동작 규칙을 정의한다.

본 문서는 기능 설계 문서이며, API 명세나 구현 상세를 확정하는 문서가 아니다.

## 기능 목적
- 인증된 사용자가 자신의 인증 상태를 명시적으로 종료할 수 있어야 한다.
- 로그아웃 요청 시 해당 계정의 모든 활성 세션을 서버 기준으로 폐기할 수 있어야 한다.
- 로그아웃 이후에는 기존에 발급된 인증 수단으로 보호 기능에 계속 접근할 수 없어야 한다.
- 로그아웃 처리와 세션 폐기 이력은 추적 가능해야 한다.

---

## 범위
이 문서는 로그아웃 기능 자체만 다룬다.

포함 범위:
- 인증된 사용자의 로그아웃 요청
- 계정 단위 전체 세션 무효화
- 액세스 토큰 및 리프레시 토큰 재사용 차단 기준
- 로그아웃 성공/실패 처리
- 로그아웃 이벤트 추적 요구사항

제외 범위:
- 로그인
- 회원가입
- 비밀번호 재설정
- 토큰 재발급 API 상세
- 관리자 강제 로그아웃
- 다중 인증 요소(2차 인증)
- 상세 UI 디자인

---

## 사용자 및 전제
- 로그아웃 대상은 이미 인증된 Personal Mail Server 사용자 계정이다.
- 현재 프로젝트는 로그인 시 액세스 토큰과 리프레시 토큰을 발급하는 구조를 사용한다.
- 로그아웃은 클라이언트 단순 정리가 아니라 서버가 계정의 전체 세션을 폐기하는 기능이어야 한다.
- 현재 저장소에는 로그아웃 구현이 없으므로, 본 문서는 후속 구현과 API 명세의 기준 문서 역할을 한다.

---

## 확정된 세부 조건
- 로그아웃 API는 `POST /api/v1/auth/logout`을 사용한다.
- 로그아웃 요청 인증은 `Authorization: Bearer <accessToken>` 형식을 사용한다.
- 로그아웃 성공 응답은 `204 No Content`를 사용한다.
- 인증 없음, 만료, 위조, 이미 무효화된 세션 요청은 모두 `401 Unauthorized`로 처리한다.
- 로그아웃 직후 기존 액세스 토큰 차단 방식은 세션 버전 관리 방식을 사용한다.
- 로그아웃 성공 후 프론트엔드는 인증 정보를 정리하고 로그인 화면(`/`)으로 이동한다.

---

## 로그아웃 기본 정의
- 로그아웃은 사용자가 명시적으로 인증 상태 종료를 요청하는 행위이다.
- 로그아웃 성공 시 서버는 해당 계정에 연결된 모든 활성 세션을 무효화해야 한다.
- 여기서 전체 세션은 동일 계정으로 발급된 모든 유효 리프레시 토큰과 그에 연결된 인증 상태를 의미한다.
- 이미 발급된 액세스 토큰 역시 로그아웃 이후 보호 기능 접근에 사용할 수 없어야 한다.

### 설계 의도
- 사용자가 한 기기에서 로그아웃하더라도 계정 기준으로 인증 상태를 명확히 정리할 수 있어야 한다.
- 금융권 수준 운영 기준에 맞게 토큰 단순 폐기가 아니라 서버 통제 가능한 세션 종료 모델을 사용해야 한다.
- 로그아웃 이후 잔존 세션이나 잔존 토큰으로 인해 인증 상태가 유지되는 모호한 상황을 방지해야 한다.

---

## 세션 무효화 원칙
- 로그아웃 성공 시 계정 기준 모든 활성 세션은 즉시 무효 상태로 전환되어야 한다.
- 무효화 대상에는 리프레시 토큰뿐 아니라 기존 액세스 토큰의 후속 사용 차단 기준도 포함되어야 한다.
- 로그아웃 직후 동일 계정의 보호 API 요청은 재인증 없이는 성공해서는 안 된다.
- 로그아웃 요청이 성공하면 후속 토큰 재발급도 허용되지 않아야 한다.

### 구현 전제 메모
- 현재 코드는 JWT 발급만 존재하고 세션 저장/폐기 구조가 없으므로, 실제 구현 시 세션 식별 또는 계정 단위 세션 버전 관리가 추가로 필요하다.
- 본 문서는 구현 방식을 고정하지 않지만, 서버가 전체 세션 무효화를 강제할 수 있는 구조를 전제로 한다.

### 구현 확정 메모
- 계정 단위 전체 세션 무효화는 세션 버전 관리 방식으로 강제한다.
- 로그아웃 성공 시 응답 본문은 사용하지 않는다.

---

## 로그아웃 처리 흐름

### 정상 흐름
1. 인증된 사용자가 로그아웃을 요청한다.
2. 시스템은 요청 사용자의 인증 상태와 계정 식별 가능 여부를 확인한다.
3. 시스템은 해당 계정의 활성 세션 전체를 무효화한다.
4. 시스템은 로그아웃 이벤트를 기록한다.
5. 시스템은 로그아웃 성공 결과를 반환한다.
6. 클라이언트는 보유 중인 토큰과 인증 상태 표시를 제거한다.

### 실패 흐름
- 인증 정보가 없거나 유효하지 않으면 로그아웃은 실패해야 한다.
- 서버가 계정을 식별할 수 없으면 로그아웃은 실패해야 한다.
- 세션 폐기 처리 중 내부 오류가 발생하면 로그아웃은 실패해야 한다.
- 실패 시에는 세션 폐기 성공으로 간주해서는 안 된다.

---

## 보안 및 품질 요구사항
- 로그아웃은 인증된 요청만 허용해야 한다.
- 로그아웃 성공 이후에는 이전 인증 수단으로 보호 자원 접근이 차단되어야 한다.
- 세션 무효화 처리 결과는 추적 가능해야 한다.
- 로그아웃 실패 응답은 민감한 내부 구현 정보를 노출해서는 안 된다.
- 로그아웃 기능은 정상, 인증 실패, 내부 오류, 중복 요청 상황을 포함해 검증 가능해야 한다.

## 수용 기준

### 정상 케이스
- 유효한 인증 상태에서 로그아웃 요청 시 성공해야 한다.
- 로그아웃 성공 시 해당 계정의 전체 세션이 무효화되어야 한다.
- 로그아웃 성공 후 기존 토큰으로 보호 API 접근이 차단되어야 한다.

### 인증 실패 케이스
- 인증 수단이 없거나 유효하지 않은 경우 실패해야 한다.
- 인증 없음, 만료, 위조, 이미 무효화된 세션 요청은 모두 `401 Unauthorized`여야 한다.

### 상태 전이 케이스
- 로그아웃 직후 동일 계정으로 토큰 재발급이 허용되어서는 안 된다.
- 로그아웃 후 다시 사용하려면 재로그인을 통해 새 인증 상태를 발급받아야 한다.
- 여러 기기에서 로그인된 상태였다면 로그아웃 후 모든 기기가 재인증 상태로 돌아가야 한다.

### 범위 제외 확인
- 본 로그아웃 기능 설계에는 관리자 강제 로그아웃이 포함되지 않아야 한다.
- 본 로그아웃 기능 설계에는 토큰 재발급 API 상세가 포함되지 않아야 한다.
- 본 로그아웃 기능 설계에는 비밀번호 재설정이 포함되지 않아야 한다.

## 정리
로그아웃 기능은 인증된 사용자가 자신의 인증 상태를 종료할 때, 서버가 해당 계정의 전체 세션을 기준으로 모든 활성 인증 상태를 폐기하는 기능이다.

또한 본 기능은 클라이언트 단순 토큰 삭제가 아니라 서버 통제가 가능한 세션 무효화 모델을 전제로 하며, 로그아웃 이후 기존 토큰은 보호 기능 접근이나 토큰 재발급에 사용될 수 없어야 한다.

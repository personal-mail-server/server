# 로그아웃 API 명세

## 문서 목적
이 문서는 Personal Mail Server의 로그아웃 기능에 대한 API 계약을 정의한다.

본 문서는 `doc/auth/logout.md`의 기능 설계를 바탕으로 작성되며, 로그아웃 요청과 응답, 인증 규칙, 상태 코드, 전체 세션 무효화 기준을 명확히 하는 것을 목적으로 한다.

포함 범위:

- 인증 헤더 사용 규칙

제외 범위:

## 엔드포인트 정의

### 로그아웃
- Method: `POST`
- Path: `/api/v1/auth/logout`
- 설명: 인증된 사용자의 로그아웃 요청을 받아 해당 계정의 전체 세션을 서버 기준으로 무효화하는 API가 필요하다.

---

## 요청 명세

### Request Headers
- `Authorization: Bearer <accessToken>`

### Request Body
- 요청 바디는 사용하지 않는다.

### Request Rules
- 로그아웃 요청은 인증된 사용자를 식별할 수 있어야 한다.
- 액세스 토큰을 기준으로 사용자를 식별해야 한다.
- 요청 본문 없이 `Authorization` 헤더만으로 처리해야 한다.

---

## 성공 응답 명세

### Success Status
- `204 No Content`

### Success Body
- 성공 응답 본문은 없다.

### Success Field Rules
- 응답 본문 필드는 사용하지 않는다.
- 성공 여부는 HTTP 상태 코드 `204 No Content`로만 판단한다.

---

## 실패 응답 명세

### 공통 원칙
- 실패 응답은 내부 구현 정보나 민감한 보안 정보를 노출하지 않아야 한다.
- 실패 시 세션 무효화 성공으로 간주해서는 안 된다.
- 실패 응답은 계정 존재 여부나 세션 저장 구조를 과도하게 노출하지 않아야 한다.

### Error Response Body
```json
{
  "code": "string",
  "message": "string"
}
```

### 상태 코드 및 에러 코드

#### `401 Unauthorized`
인증 실패 또는 무효 세션

에러 코드 예시:
- `INVALID_ACCESS_TOKEN`

응답 예시:
```json
{
  "code": "INVALID_ACCESS_TOKEN",
  "message": "인증 정보가 유효하지 않습니다. 다시 로그인해 주세요."
}
```

#### `500 Internal Server Error`
서버 내부 오류

에러 코드 예시:
- `INTERNAL_SERVER_ERROR`

응답 예시:
```json
{
  "code": "INTERNAL_SERVER_ERROR",
  "message": "요청을 처리할 수 없습니다. 잠시 후 다시 시도해 주세요."
}
```

## 전체 세션 무효화 규칙
- 로그아웃 성공 시 요청 사용자의 계정에 연결된 모든 활성 세션은 무효화되어야 한다.
- 전체 세션에는 모든 유효 리프레시 토큰과 후속 인증 상태가 포함되어야 한다.
- 로그아웃 직후 기존 액세스 토큰은 보호 API 접근에 사용할 수 없어야 한다.
- 로그아웃 직후 기존 리프레시 토큰은 토큰 재발급에 사용할 수 없어야 한다.
- 세션 무효화 범위는 단일 디바이스가 아니라 계정 전체 기준이어야 한다.

---

## 인증 헤더 사용 규칙
- 로그아웃 API는 `Authorization` 헤더의 `Bearer <accessToken>` 형식을 사용한다.
- 서버는 액세스 토큰에서 사용자 계정과 세션 버전을 식별할 수 있어야 한다.
- 요청 성공 후 액세스 토큰과 리프레시 토큰은 세션 버전 관리 기준으로 무효화되어야 한다.

---

## 구현 제약 메모
- 현재 저장소에는 토큰 발급만 존재하고 세션 무효화 저장 구조가 없다.
- 따라서 실제 구현 시에는 계정 단위 전체 세션 무효화를 강제할 수 있는 세션 버전 관리 구조가 추가되어야 한다.
- 본 문서는 그 구현 방식 자체를 고정하지 않지만, API 계약상 전체 세션 무효화 결과는 반드시 보장되어야 한다.

---

## Swagger 반영 기준

- 성공 응답은 `204 No Content` 기준으로 반영해야 한다.

## 검증 기준

### 정상 케이스
- 유효한 인증 상태로 요청하면 성공해야 한다.
- 성공 시 `204 No Content`를 반환해야 한다.
- 성공 후 동일 계정의 기존 세션은 모두 무효 상태가 되어야 한다.

### 인증 실패 케이스
- 인증 수단이 없으면 실패해야 한다.
- 위조되었거나 만료된 인증 수단이면 실패해야 한다.
- 이미 무효 상태인 세션 요청은 `401 Unauthorized`를 반환해야 한다.

### 상태 전이 케이스
- 로그아웃 성공 후 보호 API 호출은 인증 실패로 처리되어야 한다.
- 로그아웃 성공 후 토큰 재발급 요청은 인증 실패로 처리되어야 한다.
- 다중 기기 로그인 상태였다면 다른 기기의 세션도 모두 무효화되어야 한다.

### 범위 제외 확인
- 본 문서에는 로그인 API가 포함되지 않아야 한다.
- 본 문서에는 관리자 강제 로그아웃 API가 포함되지 않아야 한다.
- 본 문서에는 토큰 재발급 API가 포함되지 않아야 한다.

## 정리
로그아웃 API는 `POST /api/v1/auth/logout` 단일 엔드포인트를 통해 `Authorization: Bearer <accessToken>` 인증으로 요청을 받고, 성공 시 `204 No Content`, 인증 실패 시 `401 Unauthorized`, 내부 오류 시 `500 Internal Server Error`를 반환해야 한다.

## 필드 정의

### 1. id
- 의미: 테스트용 메일 주소 내부 식별자
- 역할: 시스템 내부 참조용 기본 키
- 요구사항:
  - 고유해야 한다.
  - 외부 노출 가능 여부는 후속 API 문서에서 결정한다.

### 2. owner_user_id
- 의미: 해당 테스트용 메일 주소를 소유한 사용자 식별자
- 역할: 사용자별 데이터 격리와 권한 검증의 기준
- 요구사항:
  - 기존 사용자 테이블과 연결되어야 한다.
  - null 이면 안 된다.

### 3. email
- 의미: 실제 테스트에 사용할 메일 주소 문자열
- 역할: 핵심 비즈니스 값
- 요구사항:
  - null 이면 안 된다.
  - 시스템 기준 유니크해야 한다.
  - 생성과 수정 모두 최종 저장 시점에 유니크 보장이 필요하다.
  - 형식 규칙의 세부 기준은 후속 API 문서에서 확정하되, 이메일로 해석 가능한 기본 형식을 만족해야 한다.

### 4. created_at
- 의미: 자원 생성 시각
- 역할: 생성 이력 추적
- 요구사항:
  - null 이면 안 된다.
  - 생성 이후 변경되면 안 된다.

### 5. updated_at
- 의미: 마지막 수정 시각
- 역할: 변경 이력 추적
- 요구사항:
  - null 이면 안 된다.
  - 수정 시 갱신되어야 한다.

### 6. deleted_at
- 의미: 논리 삭제 시각
- 역할: 물리 삭제 없이 삭제 상태를 표현
- 요구사항:
  - nullable 이어야 한다.
  - 값이 없으면 활성 상태로 본다.
  - 값이 있으면 논리 삭제 상태로 본다.

---

## 상태 해석 규칙
- `deleted_at IS NULL` 이면 활성 테스트용 메일 주소이다.
- `deleted_at IS NOT NULL` 이면 논리 삭제된 테스트용 메일 주소이다.
- 기본 목록 조회는 활성 데이터만 대상으로 해야 한다.
- 논리 삭제된 데이터의 상세 조회 허용 여부는 후속 API 문서에서 확정한다.

---

## 제약조건

### 기본 키
- `id` 는 기본 키여야 한다.

### 외래 키
- `owner_user_id` 는 사용자 테이블의 식별자를 참조해야 한다.

### 유니크 제약
- `email` 은 DB 레벨에서 유니크해야 한다.
- 애플리케이션 레벨 사전 조회는 보조 수단일 뿐, 최종 보장은 DB 제약이 담당해야 한다.

### null 제약
- `owner_user_id`, `email`, `created_at`, `updated_at` 는 null 이면 안 된다.
- `deleted_at` 는 null 가능해야 한다.

---

## 인덱스 고려사항
- `email` 유니크 인덱스가 필요하다.
- `owner_user_id` 기준 목록 조회를 위한 인덱스를 고려해야 한다.
- 활성 데이터 목록 조회가 빈번하면 `deleted_at` 조건을 고려한 인덱스를 검토할 수 있다.

인덱스의 정확한 형태는 DB 마이그레이션 설계 시 결정한다.

---

## 리포지토리 계약 기준
이 모델을 기반으로 최소한 다음 수준의 저장 동작을 지원할 수 있어야 한다.

- `Create`
- `GetByID`
- `GetByEmail`
- `ListByOwner`
- `Update`
- `SoftDelete`

각 메서드는 다음 원칙을 따라야 한다.
- `GetByID`, `ListByOwner` 는 기본적으로 활성 데이터 기준 동작을 우선한다.
- `GetByEmail` 은 유니크 충돌 검증에 사용할 수 있어야 한다.
- `SoftDelete` 는 물리 삭제가 아니라 `deleted_at` 설정 방식이어야 한다.

---

## 논리 삭제 원칙
- 삭제 시 레코드를 즉시 제거하지 않는다.
- 삭제 처리는 `deleted_at` 기록으로 표현한다.
- 삭제 이후 기본 목록에서 제외되어야 한다.
- 삭제된 주소 재사용 허용 여부는 아직 미확정이다.
- 따라서 현재 데이터 모델 문서 단계에서는 재사용 가능성을 전제로 설계 결정을 고정하지 않는다.

---

## 유니크 정책 메모
- 현재 기준 문서는 `email` 값의 시스템 전체 유니크를 요구한다.
- 이는 사용자 단위 유니크가 아니라 전체 시스템 단위 유니크를 의미한다.
- 따라서 서로 다른 사용자가 동일한 메일 주소를 각각 가질 수 없다.

- `SoftDelete` 이후 `deleted_at` 기록 확인
- 기본 목록 조회에서 논리 삭제 데이터 제외 확인

## 미확정 사항
- `id` 의 구체 타입(UUID, bigint 등)
- 삭제된 주소 재사용 허용 여부
- 상세 조회에서 삭제 데이터 노출 허용 여부
- `email` 정규화 전략(예: 소문자 강제 여부)

위 항목은 후속 API/정책 문서에서 확정한다.

---

## 정리
테스트용 메일 주소 모델은 사용자 소유의 유니크한 메일 주소를 저장하고, `deleted_at` 기반 논리 삭제를 지원하는 저장 모델이다.

이 모델은 후속 CRUD API의 공통 기반이 되어야 하며, 유니크 보장과 삭제 추적 가능성을 DB 기준으로 제공해야 한다.

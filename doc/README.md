# project/doc README

## 문서 목적
`project/doc`은 이 프로젝트의 기능, 정책, 구조, 요구사항, 설계 내용을 관리하는 문서 저장소이다.

이 디렉토리의 문서는 단순 참고용이 아니라, AI가 실제 구현과 수정 작업을 수행할 때 기준으로 삼는 문서이다.

AI는 개별 사용자 지시만으로 부분 구현하지 않고, 항상 `project/doc` 전체를 기준으로 현재 코드와 동기화해야 한다.

---

## 기본 원칙
- `project/doc`의 전체 문서는 프로젝트 구현의 기준이다.
- AI는 작업 전 항상 `project/doc` 전체를 확인한다.
- AI는 특정 문서 하나만 보고 작업하지 않는다.
- AI는 관련 문서들을 함께 검토하여 전체 맥락을 기준으로 판단한다.
- 코드 구현은 개별 요청보다 `project/doc`의 전체 문서 집합과의 정합성을 우선한다.

---

## 문서와 코드의 관계
- 문서는 코드보다 상위 기준이다.
- AI는 현재 코드 상태와 문서 내용을 비교하여 필요한 동기화 작업을 수행한다.
- 문서에 정의된 내용이 코드에 없으면 구현 대상이다.
- 문서와 코드가 다르면 수정 대상이다.
- 문서에서 제거된 내용이 코드에 남아 있으면 삭제 또는 정리 대상이다.

즉, AI는 문서를 기준으로 코드가 일치하도록 유지해야 한다.

---

## 동기화 원칙
AI는 작업 시 항상 문서와 코드의 차이를 아래 기준으로 식별한다.

### 1. 신규
문서에는 존재하지만 코드에는 없는 내용

예:
- 새 기능 정의
- 새 API 정의
- 새 정책 추가
- 새 도메인 규칙 추가

### 2. 수정
문서와 코드가 서로 다르게 유지되고 있는 내용

예:
- 요청/응답 스펙 변경
- 필드 구조 변경
- 처리 흐름 변경
- 권한 정책 변경

### 3. 삭제
문서에서는 제거되었지만 코드에는 아직 남아 있는 내용

예:
- 더 이상 사용하지 않는 API
- 제거된 상태값
- 폐기된 기능
- 더 이상 허용되지 않는 처리 방식

AI는 위 차이를 기준으로 구현, 수정, 삭제를 포함한 동기화 작업을 수행한다.

---

## 작업 방식
- 사용자가 특정 기능만 지시하더라도 AI는 항상 `project/doc` 전체를 먼저 확인한다.
- AI는 전체 문서와 현재 코드 상태를 비교한 뒤 작업 범위를 판단한다.
- AI는 부분 기능만 맞추는 것이 아니라 전체 문서와의 정합성을 고려해 작업한다.
- AI는 문서에 없는 기능을 임의로 추가하거나 유지하지 않는다.
- AI는 문서에 따라 제거된 내용이 있으면 코드에서도 제거 대상인지 검토한다.
- AI는 문서 간 연결 관계를 함께 고려해야 한다.
- 사용자 기능이 포함된 기능 설계 문서를 새로 작성하거나 수정할 때에는, 대응하는 프론트엔드 문서도 같은 작업 흐름 안에서 함께 작성하거나 갱신해야 한다.
- 프론트엔드가 현재 범위에 포함되지 않는 경우에는, 해당 기능 설계 문서 또는 대응 문서에서 프론트엔드 제외 사실을 명시해야 한다.
- 실제 구현 구성을 설명하는 문서가 존재하는 경우, 설계/코드/설정 변경 시 해당 구성 문서도 같은 작업 흐름 안에서 함께 갱신해야 한다.
- 실제 테스트 체계를 설명하는 문서가 존재하는 경우, 설계/코드/테스트 구조 변경 시 해당 테스트 문서도 같은 작업 흐름 안에서 함께 갱신해야 한다.

---

## 사용자 지시와 문서의 우선순위
- 사용자의 지시는 작업의 직접적인 계기이다.
- 하지만 실제 구현 기준은 `project/doc` 전체 문서이다.
- 사용자의 요청이 특정 기능만 가리키더라도, AI는 전체 문서와 충돌하지 않는지 반드시 확인해야 한다.
- 문서와 사용자 지시가 명확히 충돌하거나, 문서만으로 해석할 수 없는 경우에는 사용자에게 다시 확인한다.

---

## 문서 검토 범위
AI는 작업 전 다음을 포함하여 `project/doc` 전체를 검토해야 한다.

- 요구사항 문서
- API 명세 문서
- 도메인 설계 문서
- 정책 문서
- 흐름 문서
- 기타 구현 기준이 되는 문서

특정 기능만 수정하더라도 관련 문서 집합 전체를 함께 검토해야 한다.

---

## 기대 결과
이 디렉토리를 기준으로 AI가 작업할 때 기대하는 결과는 다음과 같다.

- 문서와 코드가 지속적으로 동기화된다.
- 부분 구현으로 인해 전체 정합성이 깨지지 않는다.
- 신규, 수정, 삭제가 누락 없이 반영된다.
- 프로젝트 전체가 문서 기준으로 일관되게 유지된다.

---

## 예외 처리
다음과 같은 경우에는 AI가 단정하지 않고 사용자에게 확인해야 한다.

- 문서 간 내용이 충돌하는 경우
- 문서 내용이 불완전한 경우
- 문서 해석이 여러 방향으로 가능한 경우
- 삭제인지 미반영인지 구분할 수 없는 경우
- 문서 기준만으로 구현 결정을 내리기 어려운 경우

---

## 작업 로그
- AI는 실제 변경 작업이 발생한 경우 작업 로그를 남긴다.
- 작업 로그는 `project/.ai/log` 이하에 기록한다.
- 작업 로그 규칙은 `AI_LOGGING.md`를 따른다.

---

## 문서 구조 원칙
- `project/doc`의 문서는 루트에 단순 나열하지 않는다.
- AI는 문서의 목적과 성격에 따라 적절한 폴더 구조를 설계하고 문서를 정리한다.
- 새 문서를 만들 때 AI는 먼저 적절한 위치를 판단한 뒤 해당 디렉토리에 저장한다.
- 기존 문서 역시 예외가 아니며, 필요할 경우 AI가 이동, 분류, 재배치하여 전체 구조를 정리할 수 있다.
- 같은 성격의 문서는 같은 디렉토리에 정리한다.
- 문서 수가 증가하더라도 탐색성과 관리성이 유지되도록 AI는 전체 문서 구조를 지속적으로 정리한다.
- 문서 구조 변경 시 문서 간 관계와 참조가 깨지지 않도록 함께 검토한다.
- 기능 설계, API 명세, 프론트엔드 문서처럼 같은 기능을 설명하는 관련 문서들은 가능한 한 같은 하위 디렉토리에 함께 정리한다.
- 프로젝트 전체의 현재 구성, 실행 구조, 기술 선택 결과를 설명하는 문서는 기능 문서와 분리해 `project/` 성격의 하위 디렉토리에 정리할 수 있다.
- 프로젝트 전체의 현재 테스트 체계, 검증 방식, 실행 명령을 설명하는 문서도 기능 문서와 분리해 `project/` 성격의 하위 디렉토리에 정리할 수 있다.

## 정리
`project/doc`은 프로젝트의 기능, 정책, 구조, 설계를 정의하는 기준 문서 저장소이다.

AI는 개별 요청만 보고 작업하지 않고, 항상 `project/doc` 전체와 현재 코드를 비교하여 신규, 수정, 삭제 사항을 식별하고 동기화해야 한다.

즉, 이 디렉토리는 단순 문서 모음이 아니라, 프로젝트 전체를 지속적으로 일관되게 유지하기 위한 기준이다.

# 토큰 재발급 기능 설계

## 문서 목적
이 문서는 Personal Mail Server의 토큰 재발급 기능에 대한 요구사항과 동작 규칙을 정의한다.

본 문서는 기능 설계 문서이며, API 명세나 구현 상세를 확정하는 문서가 아니다.

---

## 기능 목적
- 액세스 토큰이 만료되었거나 만료 예정인 경우, 사용자가 재로그인 없이 인증 상태를 이어갈 수 있어야 한다.
- 재발급 요청은 리프레시 토큰을 기준으로 엄격하게 검증되어야 한다.
- 로그아웃 또는 세션 무효화 이후에는 토큰 재발급이 허용되어서는 안 된다.
- 토큰 재발급 처리와 실패 사유는 추적 가능해야 한다.

---

## 범위
이 문서는 토큰 재발급 기능 자체만 다룬다.

포함 범위:
- 리프레시 토큰 기반 인증 상태 재연장
- 토큰 재발급 성공/실패 처리
- 세션 버전 검증을 통한 무효 세션 차단
- 재발급 시 새 액세스 토큰과 새 리프레시 토큰 발급
- 재발급 이벤트 추적 요구사항

제외 범위:
- 로그인
- 로그아웃
- 비밀번호 재설정
- 회원가입
- 2차 인증
- 브라우저 영속 저장소 정책 확정
- 상세 UI 디자인

---

## 사용자 및 전제
- 토큰 재발급 대상은 이미 로그인에 성공해 액세스 토큰과 리프레시 토큰을 발급받은 사용자 계정이다.
- 현재 프로젝트는 로그인 시 액세스 토큰 30분, 리프레시 토큰 7일 기준을 사용한다.
- 로그아웃은 계정 단위 전체 세션 무효화 모델을 사용하므로, 세션 버전이 바뀐 리프레시 토큰은 재발급에 사용할 수 없어야 한다.
- 현재 저장소에는 토큰 재발급 기능 문서와 백엔드 API 구현이 반영되어 있다.

---

## 확정된 세부 조건
- 토큰 재발급 API는 `POST /api/v1/auth/token/reissue`를 사용한다.
- 요청 본문에는 `refreshToken` 문자열을 포함한다.
- 재발급 요청은 액세스 토큰 없이 리프레시 토큰만으로 처리한다.
- 재발급 성공 시 새 액세스 토큰과 새 리프레시 토큰을 함께 발급한다.
- 재발급 성공 시 직전에 사용한 리프레시 토큰은 즉시 무효화되어야 한다.
- 로그아웃되었거나 세션 버전이 달라진 토큰은 재발급에 사용할 수 없다.
- 재발급 성공 응답의 토큰 만료 시간은 로그인 성공 응답과 동일한 기준을 사용한다.

---

## 토큰 재발급 기본 정의
- 토큰 재발급은 유효한 리프레시 토큰을 제출해 새 인증 토큰 쌍을 발급받는 행위이다.
- 재발급 성공 시 기존 로그인 상태를 연장하는 새 액세스 토큰과 새 리프레시 토큰을 반환해야 한다.
- 재발급은 현재 계정의 활성 세션 상태를 다시 확인한 뒤에만 성공해야 한다.

### 설계 의도
- 액세스 토큰 만료를 이유로 사용자가 불필요하게 다시 로그인하지 않도록 한다.
- 재발급 가능 여부를 리프레시 토큰 형식, 만료 여부, 세션 버전 일치 여부로 엄격히 통제한다.
- 로그아웃 이후에도 남아 있는 리프레시 토큰으로 인증을 복구하는 상황을 방지한다.

---

## 재발급 처리 원칙
- 재발급 요청은 리프레시 토큰 전용 검증을 통과해야 한다.
- 액세스 토큰을 리프레시 토큰 자리에 제출한 경우 요청은 실패해야 한다.
- 리프레시 토큰의 사용자 식별자와 세션 버전을 기준으로 현재 계정 상태를 확인해야 한다.
- 현재 계정의 세션 버전이 토큰과 다르면 요청은 실패해야 한다.
- 재발급 성공 시 새 액세스 토큰과 새 리프레시 토큰을 함께 발급해야 한다.
- 재발급에 성공한 기존 리프레시 토큰은 즉시 다시 사용할 수 없어야 한다.
- 재발급 실패 시에는 어떤 토큰도 반환하지 않아야 한다.

### 구현 전제 메모
- 현재 인증 코드는 JWT 클레임에 `typ`, `sub`, `ver`, `exp` 정보를 포함하는 구조를 사용한다.
- 본 문서는 리프레시 토큰 검증, 세션 버전 검증, 재사용 차단을 결합한 재발급 모델을 기준으로 한다.

---

## 토큰 재발급 처리 흐름

### 정상 흐름
1. 사용자가 유효한 리프레시 토큰으로 재발급을 요청한다.
2. 시스템은 요청 본문 형식을 검증한다.
3. 시스템은 리프레시 토큰의 서명, 만료 시각, 토큰 용도(`refresh`)를 검증한다.
4. 시스템은 토큰의 사용자 식별자와 세션 버전을 기준으로 현재 계정 상태를 확인한다.
5. 시스템은 현재 세션이 유효하면 사용된 기존 리프레시 토큰을 더 이상 사용할 수 없도록 처리한다.
6. 시스템은 새 액세스 토큰과 새 리프레시 토큰을 발급한다.
7. 시스템은 재발급 성공 결과를 반환한다.

### 실패 흐름
- 요청 본문이 없거나 `refreshToken`이 누락되면 재발급은 실패해야 한다.
- 리프레시 토큰 형식이 올바르지 않으면 재발급은 실패해야 한다.
- 만료되었거나 위조된 리프레시 토큰이면 재발급은 실패해야 한다.
- 로그아웃 등으로 세션 버전이 달라진 토큰이면 재발급은 실패해야 한다.
- 실패 시 새 액세스 토큰과 새 리프레시 토큰은 발급되지 않아야 한다.

---

## 보안 및 품질 요구사항
- 토큰 재발급은 리프레시 토큰만으로 허용해야 하며 다른 인증 수단을 요구하지 않는다.
- 재발급 실패 응답은 내부 구현 정보나 계정 존재 여부를 과도하게 노출해서는 안 된다.
- 세션 무효화 이후에는 이전 리프레시 토큰으로 재발급이 성공해서는 안 된다.
- 재발급에 한 번 성공한 리프레시 토큰은 재사용되어서는 안 된다.
- 재발급 시도, 성공, 실패, 세션 버전 불일치는 추적 가능해야 한다.
- 토큰 재발급 기능은 정상, 만료, 위조, 잘못된 토큰 용도, 세션 무효화 상태를 포함해 검증 가능해야 한다.

---

## 수용 기준

### 정상 케이스
- 유효한 리프레시 토큰으로 요청하면 새 액세스 토큰과 새 리프레시 토큰이 발급되어야 한다.
- 재발급 성공 시 응답의 토큰 만료 시간 기준은 로그인과 동일해야 한다.
- 재발급에 성공한 직전 리프레시 토큰은 즉시 다시 사용할 수 없어야 한다.

### 입력 검증 케이스
- `refreshToken`이 누락되면 실패해야 한다.
- JSON 형식이 아니거나 요청 본문이 잘못되면 실패해야 한다.

### 인증 실패 케이스
- 만료된 리프레시 토큰이면 실패해야 한다.
- 위조된 리프레시 토큰이면 실패해야 한다.
- 액세스 토큰을 제출하면 실패해야 한다.
- 로그아웃 이후 세션 버전이 달라진 리프레시 토큰이면 실패해야 한다.
- 이미 한 번 사용된 리프레시 토큰으로 다시 요청하면 실패해야 한다.

### 범위 제외 확인
- 본 문서에는 로그인 자체가 포함되지 않아야 한다.
- 본 문서에는 로그아웃 자체가 포함되지 않아야 한다.
- 본 문서에는 비밀번호 재설정이 포함되지 않아야 한다.

## 정리
토큰 재발급 기능은 유효한 리프레시 토큰을 기준으로 현재 세션 상태를 다시 확인한 뒤, 새 액세스 토큰과 새 리프레시 토큰을 발급해 인증 상태를 연장하는 기능이다.

또한 이 기능은 단순 토큰 문자열 교환이 아니라 세션 버전 검증을 포함해야 하며, 로그아웃 등으로 무효화된 세션에서는 재발급이 허용되어서는 안 된다.

# 토큰 재발급 프론트엔드 문서

## 문서 목적
이 문서는 토큰 재발급 기능이 프론트엔드에서 어떤 상태 전이와 처리 흐름을 가져야 하는지 정의한다.

본 문서는 프론트엔드 동작 문서이며, 토큰 재발급 기능의 핵심 규칙은 `doc/api/v1/auth/token/reissue/design.md`, API 계약은 `doc/api/v1/auth/token/reissue/backend.md`를 기준으로 한다.

---

## 기준 문서
- 기능 설계 기준 문서: `doc/api/v1/auth/token/reissue/design.md`
- API 기준 문서: `doc/api/v1/auth/token/reissue/backend.md`
- 로그인 프론트엔드 기준 문서: `doc/api/v1/auth/login/frontend.md`
- 로그아웃 프론트엔드 기준 문서: `doc/api/v1/auth/logout/frontend.md`

---

## 범위
이 문서는 토큰 재발급과 관련된 프론트엔드 상태 처리 원칙만 다룬다.

포함 범위:
- 인증 상태 유지 중 토큰 재발급 호출 시점
- 재발급 성공 시 클라이언트 인증 상태 교체 기준
- 재발급 실패 시 비인증 전환 기준
- 현재 프론트엔드 범위에서의 제외 사항 명시

제외 범위:
- 별도 토큰 재발급 전용 화면
- 보호 화면 상세 UI
- 브라우저 영속 저장소 확정
- 백그라운드 자동 재시도 정책 세부 설계
- 다중 탭 동기화 처리

---

## 현재 상태 메모
- 현재 저장소의 프론트엔드는 로그인 단일 화면과 인증 이후 상태 요약 영역을 함께 제공한다.
- 현재 프론트엔드는 토큰을 메모리 상태로만 관리하며, 영속 저장소 기반 로그인 유지 기능은 아직 구현하지 않는다.
- 현재 저장소에는 보호 화면과 자동 재발급 흐름이 구현되어 있지 않다.
- 따라서 본 문서는 후속 프론트엔드 동작 기준을 정의하는 문서 초안 성격을 가진다.

---

## 확정된 프론트엔드 조건
- 프론트엔드는 액세스 토큰이 만료되었거나 보호 API에서 인증 만료를 확인했을 때 재발급을 시도할 수 있어야 한다.
- 재발급 요청은 `POST /api/v1/auth/token/reissue`에 `refreshToken`을 포함한 JSON 본문으로 전송한다.
- 재발급 성공 시 기존 메모리 상태의 액세스 토큰과 리프레시 토큰을 새 값으로 교체해야 한다.
- 재발급 성공 후 직전에 사용한 리프레시 토큰은 더 이상 재사용하지 않아야 한다.
- 재발급 실패 시 클라이언트 인증 정보를 모두 정리하고 로그인 화면(`/`) 기준의 비인증 상태로 전환해야 한다.

---

## 프론트엔드 동작 원칙
- 사용자가 별도 버튼으로 토큰 재발급을 직접 실행하는 UI는 현재 범위에 포함하지 않는다.
- 토큰 재발급은 인증 유지 로직 내부에서 처리하는 것을 기준으로 한다.
- 재발급 시도 중에는 중복 요청이 발생하지 않도록 동일 세션 기준 하나의 요청만 진행해야 한다.
- 재발급 성공 전까지는 실패한 보호 API 요청을 성공으로 간주해서는 안 된다.
- 재발급 실패 시 로그아웃과 동일한 수준으로 인증 상태를 정리해야 한다.

---

## 요청 흐름
1. 프론트엔드는 보호 API 호출 또는 인증 유지 시점에 현재 액세스 토큰 상태를 확인한다.
2. 액세스 토큰이 만료되었거나 인증 만료 응답을 받으면 재발급을 시도한다.
3. 프론트엔드는 메모리 상태의 `refreshToken`으로 `POST /api/v1/auth/token/reissue` 요청을 전송한다.
4. 성공 시 새 토큰 쌍으로 클라이언트 인증 상태를 교체한다.
5. 실패 시 클라이언트 인증 상태를 정리하고 로그인 화면(`/`) 기준으로 되돌린다.

---

## 응답 처리 방식

### 성공 응답
- `200 OK`일 때 성공 상태로 처리한다.
- 응답 본문의 `accessToken`, `refreshToken`, 만료 시간, `tokenType`으로 인증 상태를 갱신한다.
- 이후 보호 API 호출은 새 액세스 토큰을 사용해야 한다.

### 입력 검증 실패 응답
- `400 Bad Request`일 때 재발급 요청 자체가 잘못된 것으로 처리한다.
- 이 경우 클라이언트는 현재 인증 상태를 유지하지 말고 비인증 상태로 정리해야 한다.

### 인증 실패 응답
- `401 Unauthorized`일 때 리프레시 토큰이 더 이상 유효하지 않은 것으로 처리한다.
- 이 경우 클라이언트 인증 정보를 모두 정리한 뒤 로그인 화면(`/`) 기준으로 전환해야 한다.

### 서버 오류 응답
- `500 Internal Server Error`일 때 서버 오류로 처리한다.
- 이 경우 현재 요청은 성공으로 간주해서는 안 된다.
- 프론트엔드는 재시도 정책이 아직 확정되지 않았으므로, 사용자에게 인증 상태를 다시 확인하도록 유도하는 수준에서 처리한다.

### 네트워크 오류
- 네트워크 오류나 예기치 않은 요청 실패 시 프론트엔드 전용 오류 코드를 사용할 수 있다.
- 이 경우에도 재발급 성공으로 오인해서는 안 된다.

---

## 클라이언트 상태 정리 원칙
- 재발급 성공 시 기존 액세스 토큰, 리프레시 토큰, 만료 시간은 새 값으로 교체해야 한다.
- 재발급 성공 시 직전 리프레시 토큰 참조는 즉시 폐기해야 한다.
- 재발급 실패 시 메모리 상태의 인증 관련 값은 모두 제거해야 한다.
- 현재 저장소는 영속 저장소 기반 토큰 저장을 포함하지 않으므로, 실제 저장 위치가 확장되더라도 실패 시 전체 정리 원칙은 유지해야 한다.

---

## 검증 기준
- 인증 유지 로직에서 재발급 엔드포인트를 호출할 수 있어야 한다.
- 성공 시 새 토큰 쌍으로 인증 상태가 갱신되어야 한다.
- 성공 후 이전 리프레시 토큰이 다시 사용되지 않아야 한다.
- `401 Unauthorized` 응답 시 비인증 상태로 전환되어야 한다.
- 실패 시 이전 인증 상태를 성공으로 오인하지 않아야 한다.
- 현재 범위에서 별도 재발급 전용 화면이 없어야 한다.

---

## 현재 구현상 비포함 항목
- 재발급 자동 실행이 붙은 보호 화면 UI
- 영속 저장소 기반 로그인 유지
- 다중 탭 동기화
- 브라우저 자동화 테스트

---

## 정리
토큰 재발급 프론트엔드는 사용자가 직접 조작하는 별도 화면보다 인증 유지 로직 내부에서 동작하는 흐름을 기준으로 하며, 성공 시 새 토큰 쌍으로 인증 상태를 교체하고 실패 시 비인증 상태로 되돌려야 한다.

또한 현재 저장소에는 자동 재발급 UI와 보호 화면이 아직 없으므로, 본 문서는 후속 프론트엔드 확장 시 따라야 할 상태 전이 기준을 먼저 정리하는 역할을 가진다.

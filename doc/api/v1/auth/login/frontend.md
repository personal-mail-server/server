# 로그인 프론트엔드 문서

## 문서 목적
이 문서는 현재 구현된 로그인 웹 화면의 구조와 사용자 동작을 정의한다.

본 문서는 프론트엔드 동작 문서이며, 로그인 기능의 핵심 규칙은 `doc/api/v1/auth/login/design.md`, API 계약은 `doc/api/v1/auth/login/backend.md`를 기준으로 한다.

---

## 기준 문서
- 기능 설계 기준 문서: `doc/api/v1/auth/login/design.md`
- API 기준 문서: `doc/api/v1/auth/login/backend.md`
- 현재 구현 기준 파일:
  - `code/frontend/index.html`
  - `code/frontend/styles.css`
  - `code/frontend/app.js`

---

## 범위
이 문서는 현재 구현된 로그인 단일 화면만 다룬다.

포함 범위:
- 로그인 페이지 진입 화면
- 입력 필드 구성
- 클라이언트 입력 검증
- 로그인 요청 전송 동작
- 응답 상태 표시 방식
- 로그인 성공 후 인증 상태 전환
- Swagger 진입 링크

제외 범위:
- 회원가입 화면
- 비밀번호 재설정 화면
- 토큰 재발급 UI
- 전역 네비게이션
- 다단계 인증 UI

---

## 화면 진입점
- 로그인 프론트엔드는 프론트엔드 루트 경로(`/`)에서 접근한다.
- 사용자는 Docker 실행 후 브라우저에서 로그인 화면을 직접 확인할 수 있어야 한다.
- Swagger는 로그인 화면 내부 링크를 통해 `/docs` 경로로 접근할 수 있어야 한다.

---

## 화면 구조
현재 로그인 화면은 일반적인 단일 카드 형태를 따른다.

구성 요소:
- 제목: `로그인`
- 보조 문구: Personal Mail Server 계정 로그인 안내 문구
- 입력 필드:
  - `로그인 ID`
  - `비밀번호`
- 액션 버튼:
  - `로그인`
- 보조 정보 영역:
  - 테스트 계정 표시
  - Swagger 보기 링크
- 인증 상태 영역:
  - 로그인 성공 후 로그인 ID, 토큰 타입, 만료 시간 표시
  - 사용자 메뉴 기반 로그아웃 액션
- 상태 표시 영역:
  - 요청 결과 메시지
  - 응답 JSON 출력 영역

---

## 입력 규칙

### 로그인 ID
- 입력 필드는 텍스트 타입을 사용한다.
- 최대 길이는 32자이다.
- 제출 전 앞뒤 공백은 제거한다.
- 클라이언트 검증 규칙은 서버 문서와 동일하게 적용한다.
- 허용 형식은 영문 소문자, 숫자, 하이픈(`-`)이며 길이는 4자 이상 32자 이하이다.

### 비밀번호
- 입력 필드는 비밀번호 타입을 사용한다.
- 최대 길이는 64자이다.
- 클라이언트 검증 규칙은 서버 문서와 동일하게 적용한다.
- 길이는 8자 이상 64자 이하이며, 영문자 1개 이상과 숫자 1개 이상을 포함해야 한다.
- 공백은 허용하지 않는다.

---

## 제출 흐름
1. 사용자가 로그인 ID와 비밀번호를 입력한다.
2. 프론트엔드는 클라이언트 규칙으로 먼저 입력값을 검증한다.
3. 검증 실패 시 서버 요청 없이 상태 영역에 오류 메시지를 표시한다.
4. 검증 성공 시 `POST /api/v1/auth/login` 요청을 전송한다.
5. 요청 중에는 버튼을 비활성화한다.
6. 로그인 성공 시 인증 상태 영역으로 전환하고 응답 토큰 정보를 메모리 상태에 보관한다.
7. 응답 결과에 따라 상태 영역을 갱신한다.
8. 요청 완료 후 버튼을 다시 활성화한다.

---

## 응답 처리 방식

### 성공 응답
- `200 OK`일 때 성공 상태로 표시한다.
- 상태 영역에 성공 메시지를 보여준다.
- 응답 본문의 JSON을 그대로 출력한다.
- 따라서 액세스 토큰, 리프레시 토큰, 만료 시간, 토큰 타입이 화면에서 확인 가능하다.
- 로그인 성공 후 인증 상태 요약 영역과 사용자 메뉴 기반 로그아웃 액션이 보여야 한다.

### 잠금 응답
- `423 Locked`일 때 경고 상태로 표시한다.
- 잠금 메시지를 사용자에게 보여준다.
- 응답 본문의 JSON을 그대로 출력한다.

### 일반 실패 응답
- `400`, `401`, `500` 등의 응답은 오류 상태로 표시한다.
- 서버가 내려준 `message` 값을 우선 표시한다.
- 응답 본문의 JSON을 그대로 출력한다.

### 네트워크 오류
- 네트워크 오류나 예기치 않은 요청 실패 시 프론트엔드 전용 오류 코드를 사용한다.
- 이 경우 상태 영역에는 `NETWORK_ERROR`와 브라우저가 제공한 오류 메시지를 표시한다.

---

## 현재 구현상 비포함 항목
- 브라우저 영속 저장소(localStorage, sessionStorage, cookie) 기반 토큰 저장은 아직 구현하지 않는다.
- 별도 보호 페이지 라우팅은 아직 구현하지 않는다.
- 리프레시 토큰 재발급 흐름은 없다.

---

## 검증 기준
- 로그인 화면은 브라우저에서 루트 경로(`/`)로 열려야 한다.
- 로그인 ID와 비밀번호 입력 필드가 모두 보여야 한다.
- 잘못된 입력 형식은 서버 요청 전에 차단되어야 한다.
- 유효한 입력은 `POST /api/v1/auth/login`으로 전송되어야 한다.
- 로그인 성공 시 응답 JSON이 상태 영역에 표시되어야 한다.
- 로그인 성공 시 인증 상태 요약 영역이 표시되어야 한다.
- 잠금 상태 응답 시 경고 상태가 보여야 한다.
- 실패 응답 시 오류 상태가 보여야 한다.
- Swagger 링크를 통해 `/docs`에 접근 가능해야 한다.

---

## 정리
현재 로그인 프론트엔드는 로그인 입력 화면과 인증 이후 상태 표시 영역을 함께 제공하며, 일반적인 카드형 로그인 폼에서 입력 검증, 요청 전송, 응답 상태 확인이 가능하도록 구성되어 있다.

또한 이 화면은 로그인 성공/실패/잠금 상태와 로그아웃 진입점을 웹에서 직접 확인하기 위한 최소 구현이며, 별도 보호 페이지 라우팅이나 토큰 재발급 흐름은 아직 포함하지 않는다.

# 로그인 기능 설계

## 문서 목적
이 문서는 Personal Mail Server의 로그인 기능에 대한 요구사항과 동작 규칙을 정의한다.

본 문서는 기능 설계 문서이며, API 명세나 구현 상세를 확정하는 문서가 아니다.

---

## 기능 목적
- 사용자가 시스템에 안전하게 로그인할 수 있어야 한다.
- 로그인 성공 후 인증된 사용자 상태를 유지할 수 있어야 한다.
- 반복 실패나 비정상 시도를 통제할 수 있어야 한다.
- 운영 기준에 맞게 추적 가능하고 검증 가능한 로그인 흐름을 가져야 한다.

---

## 범위
이 문서는 로그인 기능 자체만 다룬다.

포함 범위:
- 로그인 ID와 비밀번호를 이용한 인증
- 로그인 성공/실패 처리
- 실패 횟수 누적 및 계정 잠금 정책
- 액세스 토큰과 리프레시 토큰 발급
- 로그인 상태 유지를 위한 토큰 구조 사용

제외 범위:
- 로그아웃
- 비밀번호 재설정
- 이메일 인증
- 2차 인증
- 회원가입
- 상세 API 명세
- 화면 UI 설계

---

## 사용자 및 전제
- 로그인 대상은 Personal Mail Server를 사용하는 사용자 계정이다.
- 사용자는 사전에 발급되거나 생성된 로그인 ID와 비밀번호를 보유하고 있어야 한다.
- 인증 이후 사용자는 보호된 기능에 접근할 수 있어야 한다.

---

## 로그인 식별자 규칙

### 기본 정의
- 로그인 ID는 이메일이 아니다.
- 로그인 ID는 영문 소문자 기반의 평문 식별자이다.

### 허용 규칙
- 길이: 4자 이상 32자 이하
- 허용 문자: 영문 소문자, 숫자, 하이픈(`-`)
- 공백은 허용하지 않는다.
- 대문자는 허용하지 않는다.
- 언더스코어(`_`)는 허용하지 않는다.
- 하이픈 외 특수문자는 허용하지 않는다.

### 설계 의도
- 사람이 기억하고 직접 입력할 수 있는 간단한 식별자를 유지한다.
- 이메일 형식과 분리해 로그인 식별자의 의미를 명확히 한다.
- 허용 문자 집합을 단순화해 검증과 추적을 쉽게 한다.

---

## 비밀번호 규칙
- 길이: 8자 이상 64자 이하
- 영문자 1개 이상 포함해야 한다.
- 숫자 1개 이상 포함해야 한다.
- 특수문자는 선택적으로 사용할 수 있다.
- 공백은 허용하지 않는다.

### 비밀번호 검증 원칙
- 규칙을 만족하지 않는 비밀번호는 로그인 입력으로 허용하지 않는다.
- 비밀번호 오류 응답은 과도한 내부 정보를 노출하지 않아야 한다.
- 저장 방식, 해시 알고리즘, 비밀 관리 방식은 후속 보안 설계에서 확정한다.

---

## 인증 방식
- 인증 방식은 액세스 토큰과 리프레시 토큰 구조를 사용한다.
- 로그인 성공 시 서버는 액세스 토큰과 리프레시 토큰을 발급해야 한다.
- 액세스 토큰은 보호된 기능 접근에 사용한다.
- 리프레시 토큰은 로그인 상태 유지를 위한 토큰 재발급 흐름에 사용한다.

### 설계 원칙
- 토큰 발급 및 검증은 추적 가능해야 한다.
- 토큰 수명, 저장 방식, 재발급 정책, 폐기 정책은 후속 문서에서 구체화한다.
- 본 문서에서는 로그인 성공 시 토큰 쌍을 발급한다는 기능 요구만 확정한다.

---

## 로그인 처리 흐름

### 정상 흐름
1. 사용자가 로그인 ID와 비밀번호를 입력한다.
2. 시스템은 입력 형식을 검증한다.
3. 시스템은 계정 존재 여부와 인증 가능 상태를 확인한다.
4. 시스템은 비밀번호를 검증한다.
5. 시스템은 인증 성공 시 실패 횟수를 초기화한다.
6. 시스템은 액세스 토큰과 리프레시 토큰을 발급한다.
7. 시스템은 인증 성공 결과를 반환한다.

### 실패 흐름
- 로그인 ID 형식이 유효하지 않으면 로그인은 실패해야 한다.
- 존재하지 않는 계정이면 로그인은 실패해야 한다.
- 비밀번호가 일치하지 않으면 로그인은 실패해야 한다.
- 잠금 상태 계정이면 로그인은 실패해야 한다.
- 실패 시 토큰은 발급되지 않아야 한다.

---

## 로그인 실패 제한 및 잠금 정책
- 로그인 연속 실패 5회 시 계정은 10분 동안 잠금 상태가 되어야 한다.
- 실패 횟수는 연속 실패 기준으로 누적한다.
- 로그인 성공 시 누적 실패 횟수는 초기화되어야 한다.
- 잠금 기간 중에는 올바른 비밀번호를 입력하더라도 로그인할 수 없어야 한다.
- 잠금 해제 시점이 지나면 다시 로그인 시도를 허용해야 한다.

### 운영 원칙
- 잠금 상태와 실패 횟수 변화는 추적 가능해야 한다.
- 인증 실패 메시지는 계정 존재 여부를 과도하게 추론할 수 없도록 설계해야 한다.
- 보안상 필요한 경우 운영자가 잠금 상태를 확인할 수 있어야 한다.

---

## 로그인 유지
- 로그인 유지는 리프레시 토큰을 기반으로 지원하는 것을 전제로 한다.
- 사용자는 액세스 토큰 만료 후에도 유효한 리프레시 토큰이 있으면 인증 상태를 이어갈 수 있어야 한다.
- 실제 재발급 절차와 만료 정책은 후속 API 명세 또는 인증 설계 문서에서 확정한다.

---

## 보안 및 품질 요구사항
- 로그인 입력값은 명시적으로 검증되어야 한다.
- 인증 실패, 잠금, 토큰 발급 같은 핵심 이벤트는 추적 가능해야 한다.
- 내부 오류가 발생해도 민감한 정보가 외부로 노출되어서는 안 된다.
- 로그인 기능은 정상, 실패, 예외, 경계값을 포함해 검증 가능해야 한다.
- 보안성과 통제 가능성이 편의성보다 우선한다.

---

## 수용 기준

### 정상 케이스
- 유효한 로그인 ID와 올바른 비밀번호를 입력하면 로그인에 성공해야 한다.
- 로그인 성공 시 액세스 토큰과 리프레시 토큰이 함께 발급되어야 한다.
- 로그인 성공 시 기존 실패 횟수는 초기화되어야 한다.

### 입력 검증 케이스
- 4자 미만 또는 32자 초과 로그인 ID는 허용되지 않아야 한다.
- 영문 소문자, 숫자, 하이픈 외 문자가 포함된 로그인 ID는 허용되지 않아야 한다.
- 영문자가 없거나 숫자가 없는 비밀번호는 허용되지 않아야 한다.
- 8자 미만 또는 64자 초과 비밀번호는 허용되지 않아야 한다.
- 공백이 포함된 비밀번호는 허용되지 않아야 한다.

### 실패 및 잠금 케이스
- 잘못된 비밀번호 입력 시 실패 횟수가 누적되어야 한다.
- 연속 5회 실패 시 계정이 10분 잠금 상태가 되어야 한다.
- 잠금 기간 중 로그인은 항상 실패해야 한다.
- 잠금 해제 시점 이후에는 다시 로그인 시도가 가능해야 한다.

### 범위 제외 확인
- 본 로그인 기능 설계에는 로그아웃이 포함되지 않아야 한다.
- 본 로그인 기능 설계에는 비밀번호 재설정이 포함되지 않아야 한다.
- 본 로그인 기능 설계에는 이메일 인증이 포함되지 않아야 한다.
- 본 로그인 기능 설계에는 2차 인증이 포함되지 않아야 한다.

---

## 테스트 관점 체크포인트
- 정상 로그인 성공
- 존재하지 않는 계정 로그인 실패
- 잘못된 비밀번호 로그인 실패
- 로그인 ID 형식 오류
- 비밀번호 형식 오류
- 연속 실패에 따른 잠금 전환
- 잠금 해제 이후 재로그인 허용
- 로그인 성공 후 실패 횟수 초기화

---

## 후속 문서화 필요 항목
- 로그인 API 명세
- 토큰 재발급 정책
- 비밀번호 저장 정책
- 감사 로그 상세 항목
- 관리자 관제 또는 계정 잠금 해제 절차

## 정리
로그인 기능은 이메일이 아닌 영문 소문자 기반 ID와 비밀번호를 사용해 사용자를 인증하고, 성공 시 액세스 토큰과 리프레시 토큰을 발급하는 기능이다.

또한 연속 실패 5회 시 10분 잠금 정책을 적용하며, 로그인 유지 기능은 포함하되 로그아웃, 비밀번호 재설정, 이메일 인증, 2차 인증은 현재 범위에서 제외한다.

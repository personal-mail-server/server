# PR 1 - 테스트용 메일 주소 데이터 모델과 저장 기반

## PR 목적
- 이후 생성/조회/수정/삭제 API가 공통으로 사용할 테스트용 메일 주소 저장 기반을 먼저 확정한다.
- 이 PR 하나만 머지되어도 DB 스키마, 내부 모델, 저장 규칙, 테스트 기반이 준비된 상태여야 한다.

## 목표
- 테스트용 메일 주소 관리 기능의 DB 기반과 내부 저장 구조를 먼저 만든다.

## 선행 조건
- 기존 인증 사용자 테이블과 사용자 식별 체계가 이미 존재해야 한다.
- 현재 프로젝트의 DB 마이그레이션 체계는 `code/migrations/` 기반 SQL 업/다운 파일 구조를 따라야 한다.

## 포함 범위
- 테스트용 메일 주소 테이블 추가
- 소유 사용자 연결 필드 추가
- `deleted_at` 기반 논리 삭제 구조 추가
- 유니크 제약 추가
- 기본 리포지토리/저장 계층 추가

## 구현 대상 파일 범위
- 마이그레이션
  - `code/migrations/*_create_test_mail_addresses.sql`
  - `code/migrations/*_create_test_mail_addresses.down.sql`
- 내부 모델/리포지토리
  - `code/internal/...` 아래 테스트용 메일 주소 도메인 패키지
- 테스트
  - 리포지토리 또는 DB 통합 테스트 파일
- 문서
  - 데이터 모델 문서
  - 필요 시 논리 삭제 정책 초안

## 제외 범위
- HTTP API
- OpenAPI
- 프론트엔드
- 후보 메일 생성 규칙
- 입력 검증 규칙의 최종 확정

## 스키마 요구사항
- 테스트용 메일 주소 테이블은 최소한 다음 정보를 저장할 수 있어야 한다.
  - 내부 식별자
  - 소유 사용자 식별자
  - 메일 주소 값
  - 생성 시각
  - 수정 시각
  - 삭제 시각(`deleted_at`)
- 메일 주소 값은 시스템 기준 유니크하게 관리되어야 한다.
- 논리 삭제를 지원하기 위한 `deleted_at` 필드는 nullable 이어야 한다.
- 소유 사용자 식별자는 기존 사용자 테이블과 연결되어야 한다.
- 생성/수정 시각은 추적 가능해야 한다.

## 데이터 모델 원칙
- 내부 모델은 API DTO와 분리된 저장 모델이어야 한다.
- 논리 삭제 상태를 모델에서 명시적으로 표현할 수 있어야 한다.
- 후속 PR에서 목록/상세/생성/수정/삭제에 모두 재사용 가능한 최소 필드를 가져야 한다.

## 리포지토리 책임 범위
- 테스트용 메일 주소 저장
- 메일 주소 유니크 충돌 감지에 필요한 기본 조회
- 소유자 기준 단건 조회
- 기본 활성 데이터 조회를 위한 논리 삭제 제외 처리 기반
- 논리 삭제 처리를 위한 업데이트 기반

## 이번 PR에서 허용되는 리포지토리 메서드 수준
- `Create`
- `GetByID`
- `GetByEmail`
- `ListByOwner`
- `Update`
- `SoftDelete`

단, 이 PR에서는 HTTP 레이어 없이 저장 계층 계약과 테스트 가능성 확보가 목적이다.

## 문서 범위

- 논리 삭제 정책 초안 추가 가능

## 테스트 범위
- 저장 성공
- 유니크 제약 검증
- 논리 삭제 컬럼 기본 동작

## 필수 테스트 케이스
- 새 테스트용 메일 주소 저장 성공
- 동일 메일 주소 중복 저장 실패
- 서로 다른 사용자라도 동일 메일 주소 저장 실패
- 소유자 기준 단건 조회 성공
- 논리 삭제 수행 시 `deleted_at` 기록 성공
- 기본 목록 조회에서 논리 삭제 데이터 제외 확인

## 비포함 테스트
- HTTP 요청/응답 테스트
- OpenAPI 검증 테스트
- 프론트엔드 테스트

## 머지 이후 기대 상태
- DB에 테스트용 메일 주소 저장 구조가 존재한다.
- 논리 삭제 가능한 기본 모델이 존재한다.
- 후속 API PR은 저장 계층을 새로 설계하지 않고 바로 사용할 수 있다.

## 머지 조건
- 이후 API PR들이 공통으로 사용할 저장 기반이 안정적으로 준비되어야 한다.

## 리뷰 체크리스트
- 물리 삭제가 아닌 논리 삭제 구조인지
- 유니크 제약이 DB 레벨에서 보장되는지
- 사용자 소유권 연결이 가능한 구조인지
- 후속 PR이 API 레이어만 추가하면 되도록 저장 책임이 분리되었는지

# 테스트용 메일 주소 관리 요구사항 문서

## 문서 목적
이 문서는 Personal Mail Server의 1차 구현 범위를 테스트용 메일 주소 관리 기능으로 축소해 요구사항을 정의한다.

본 문서는 메일 서버 전체 범위를 다루지 않고, 테스트용 메일 주소의 목록, 상세, 생성, 수정, 삭제와 유니크 메일 생성/검증에 한정한 요구사항 문서이다.

## 배경
- 현재 프로젝트는 인증 기반과 실행 환경은 갖추고 있지만, 메일 도메인 기능은 아직 본격적으로 정의되지 않았다.
- 메일 서버 전체를 한 번에 설계하고 구현하기보다, 먼저 테스트용 메일 주소를 안전하게 관리하는 기능부터 확정할 필요가 있다.
- 사용자는 테스트를 위해 여러 메일 주소를 준비해야 하지만, 직접 유니크한 값을 매번 고민하지 않고도 사용할 수 있어야 한다.

---

## 목표
- 인증된 사용자가 자신의 테스트용 메일 주소를 목록과 상세로 확인할 수 있어야 한다.
- 사용자가 테스트용 메일 주소를 생성, 수정, 삭제할 수 있어야 한다.
- 삭제는 물리 삭제가 아니라 논리 삭제 방식이어야 한다.
- 생성 및 수정 시 메일 주소의 유니크 여부를 검증해야 한다.
- 사용자가 직접 메일 주소를 입력할 수 있어야 한다.
- 동시에 서버가 유니크한 메일 주소 후보를 반환하는 API도 제공해야 한다.

## 1차 범위(MVP)

### 포함 범위
- 인증된 사용자 기준 테스트용 메일 주소 목록 조회
- 인증된 사용자 기준 테스트용 메일 주소 상세 조회
- 테스트용 메일 주소 생성
- 테스트용 메일 주소 수정
- 테스트용 메일 주소 삭제
- 메일 주소 유니크 여부 검증
- 서버가 유니크 메일 주소 후보를 반환하는 API 제공
- 웹 UI에서 위 기능을 확인할 수 있는 최소 흐름

### 제외 범위
- 메일 수신 기능
- 메일 저장 및 메일 조회 기능
- 반송, 수신 거부, 스팸 처리 정책
- 메일 주소 그룹 기능
- 외부 메일 발송 기능
- SMTP 송수신 인프라 전체
- 관리자 기능

### 핵심 제약
- 현재 범위의 중심은 메일 자체가 아니라 테스트용 메일 주소 자원 관리이다.
- 삭제는 데이터베이스 물리 삭제가 아니라 논리 삭제여야 한다.
- 유니크 주소 생성 API는 사용 편의 기능이지만, 사용자의 직접 입력 가능성을 대체해서는 안 된다.

## 사용자 가치
- 사용자는 테스트용 메일 주소를 한 곳에서 쉽게 관리할 수 있어야 한다.
- 사용자는 직접 주소를 입력할 수도 있고, 서버가 추천한 유니크 주소를 바로 사용할 수도 있어야 한다.
- 삭제된 주소의 이력을 운영상 추적할 수 있어야 한다.

---

## 주요 사용자 시나리오

### 시나리오 1: 목록 확인
1. 사용자가 로그인한다.
2. 사용자가 테스트용 메일 주소 목록 화면에 진입한다.
3. 사용자는 현재 활성 주소 목록을 확인한다.

### 시나리오 2: 직접 입력 생성
1. 사용자가 새 메일 주소 생성 화면에 진입한다.
2. 사용자가 원하는 메일 주소를 직접 입력한다.
3. 시스템은 해당 메일 주소가 유니크한지 검증한다.
4. 유니크하면 생성이 완료된다.

### 시나리오 3: 자동 생성 사용
1. 사용자가 새 메일 주소 생성 화면에 진입한다.
2. 사용자가 서버가 생성한 유니크 메일 주소 후보를 요청한다.
3. 시스템은 현재 사용 가능한 유니크 메일 주소를 반환한다.
4. 사용자는 해당 값을 그대로 사용하거나 일부 수정 후 생성할 수 있다.

### 시나리오 4: 수정
1. 사용자가 기존 메일 주소 상세 화면에 진입한다.
2. 사용자가 메일 주소 또는 관련 메타데이터를 수정한다.
3. 시스템은 수정 결과가 유니크 조건을 만족하는지 검증한다.
4. 조건을 만족하면 수정이 완료된다.

### 시나리오 5: 삭제
1. 사용자가 기존 메일 주소 삭제를 요청한다.
2. 시스템은 해당 자원을 논리 삭제 상태로 전환한다.
3. 일반 목록에서는 삭제된 주소가 기본적으로 노출되지 않는다.

## 핵심 기능 요구사항

### 1. 계정 기반 접근
- 테스트용 메일 주소 관리 기능은 인증된 사용자만 사용할 수 있어야 한다.
- 사용자는 자신의 메일 주소만 조회, 수정, 삭제할 수 있어야 한다.
- 다른 사용자의 메일 주소 상세 정보는 접근할 수 없어야 한다.

### 2. 테스트용 메일 주소 목록 조회
- 사용자는 자신이 생성한 테스트용 메일 주소 목록을 볼 수 있어야 한다.
- 목록에는 최소한 메일 주소, 생성 일시, 수정 일시, 삭제 여부를 식별할 수 있어야 한다.
- 기본 목록에서는 논리 삭제된 주소를 제외해야 한다.
- 삭제된 주소 조회 정책은 후속 API 명세에서 구체화할 수 있다.

### 3. 테스트용 메일 주소 상세 조회
- 사용자는 특정 테스트용 메일 주소의 상세 정보를 볼 수 있어야 한다.
- 상세에는 최소한 메일 주소 값, 생성 일시, 수정 일시, 삭제 여부를 포함해야 한다.
- 논리 삭제된 주소의 상세 조회 허용 여부는 후속 API 명세에서 확정하되, 무단 복구나 정보 노출이 발생하지 않아야 한다.

### 4. 테스트용 메일 주소 생성
- 사용자는 테스트용 메일 주소를 직접 입력해 생성할 수 있어야 한다.
- 생성 시 메일 주소 값은 필수 입력이어야 한다.
- 생성 시 메일 주소는 시스템 전체에서 유니크해야 한다.
- 이미 사용 중이거나 삭제 정책상 재사용 불가한 메일 주소는 생성할 수 없어야 한다.
- 메일 주소 형식 규칙은 후속 상세 설계에서 확정하되, 이메일로 해석 가능한 기본 형식을 만족해야 한다.

### 5. 테스트용 메일 주소 수정
- 사용자는 기존 테스트용 메일 주소를 수정할 수 있어야 한다.
- 수정 시에도 메일 주소 값의 유니크 여부를 다시 검증해야 한다.
- 자기 자신과 동일한 현재 값은 유니크 충돌로 처리되어서는 안 된다.
- 어떤 필드까지 수정 가능한지는 후속 상세 설계에서 확정할 수 있으나, 최소한 메일 주소 값 변경 가능 여부는 명시적으로 다뤄야 한다.

### 6. 테스트용 메일 주소 삭제
- 삭제는 물리 삭제가 아니라 논리 삭제 방식이어야 한다.
- 논리 삭제 시 원본 데이터는 즉시 DB에서 제거되지 않아야 한다.
- 논리 삭제된 주소는 기본 사용자 목록에서 제외되어야 한다.
- 삭제된 주소의 재사용 허용 여부는 후속 상세 설계에서 확정해야 한다.
- 삭제 처리 시 삭제 시각 등 추적 가능한 정보가 남아야 한다.

### 7. 유니크 메일 생성 API
- 서버는 현재 사용 가능한 유니크 메일 주소 후보를 반환하는 API를 제공해야 한다.
- 이 API는 사용자가 메일 주소를 일일이 만들지 않아도 되도록 돕는 목적이어야 한다.
- 반환된 값은 생성 시점 기준으로 유니크 후보여야 한다.
- 다만 최종 생성 시점에도 다시 유니크 검증을 수행해야 한다.
- 사용자는 서버가 제안한 값을 그대로 쓰지 않고 직접 수정할 수 있어야 한다.

### 8. 유니크 검증 원칙
- 생성과 수정 모두 저장 직전에 유니크 검증을 수행해야 한다.
- 단순 사전 조회만으로 유니크를 보장했다고 간주해서는 안 된다.
- 동시 요청 상황에서도 중복 메일 주소가 저장되지 않도록 보장해야 한다.

### 9. 웹 기반 사용자 흐름
- 사용자는 Docker 실행 후 웹에서 테스트용 메일 주소 목록, 상세, 생성, 수정, 삭제를 확인할 수 있어야 한다.
- 생성 화면에서는 직접 입력과 자동 생성값 사용 흐름을 모두 지원해야 한다.
- Swagger는 API 확인용으로 제공하되, 실제 사용자 기능은 웹에서도 검증 가능해야 한다.

## 비기능 요구사항

### 보안
- 인증, 권한 검증, 입력값 검증이 기본 전제여야 한다.
- 다른 사용자의 메일 주소 데이터가 노출되어서는 안 된다.

### 추적 가능성
- 생성, 수정, 삭제 시점은 추적 가능해야 한다.
- 논리 삭제 여부와 삭제 시점은 운영상 확인 가능해야 한다.

### 안정성
- 동시 생성 또는 동시 수정 상황에서도 유니크 제약이 깨지지 않아야 한다.

### 검증 가능성
- 목록, 상세, 생성, 수정, 삭제, 자동 생성, 유니크 충돌 케이스를 테스트할 수 있어야 한다.

## 수용 기준

### 목록/상세
- 로그인한 사용자는 자신의 테스트용 메일 주소 목록을 조회할 수 있어야 한다.
- 로그인한 사용자는 특정 테스트용 메일 주소의 상세를 조회할 수 있어야 한다.
- 다른 사용자의 주소는 조회할 수 없어야 한다.

### 생성
- 사용자는 직접 입력한 메일 주소로 생성할 수 있어야 한다.
- 이미 존재하는 메일 주소로는 생성할 수 없어야 한다.
- 서버가 반환한 유니크 후보값으로도 생성할 수 있어야 한다.

### 수정
- 사용자는 기존 메일 주소를 수정할 수 있어야 한다.
- 수정 결과가 다른 자원의 메일 주소와 충돌하면 실패해야 한다.

### 삭제
- 삭제 요청 시 해당 자원은 논리 삭제되어야 한다.
- 삭제 후 기본 목록에서는 노출되지 않아야 한다.
- 삭제 이력은 추적 가능해야 한다.

### 자동 생성 API
- 서버는 유니크 메일 주소 후보를 반환할 수 있어야 한다.
- 후보를 받은 뒤 사용자가 직접 수정해 저장하는 것도 가능해야 한다.

## 미확정 사항
- 테스트용 메일 주소의 도메인 규칙
- 삭제된 주소의 재사용 허용 여부
- 목록에서 삭제된 주소를 별도 필터로 조회할지 여부
- 자동 생성 메일 주소의 생성 규칙과 길이 제약
- 수정 가능 필드의 정확한 범위

미확정 사항은 후속 상세 문서에서 확정하되, 본 문서의 핵심 범위인 테스트용 메일 주소 CRUD와 논리 삭제, 유니크 검증, 자동 생성 API 요구를 훼손해서는 안 된다.

---

## 정리
현재 1차 구현 범위는 테스트용 메일 주소의 목록, 상세, 생성, 수정, 삭제를 제공하는 관리 기능이다.

삭제는 논리 삭제여야 하며, 생성과 수정에는 유니크 검증이 필요하다. 또한 사용자가 직접 메일 주소를 입력할 수 있어야 하고, 동시에 서버가 유니크 메일 주소 후보를 반환하는 API도 제공해야 한다.

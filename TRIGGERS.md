# TRIGGERS.md

## 문서 목적
이 문서는 프로젝트 전역에서 사용하는 예약된 트리거 명령어와 그 해석 규칙을 정의한다.

AI는 사용자의 입력이 이 문서에 정의된 트리거와 일치할 경우, 일반적인 질의응답이 아니라 사전에 정의된 작업 흐름으로 해석해야 한다.

## 기본 원칙
- 트리거는 프로젝트 전역에서 사용하는 예약된 작업 지시어이다.
- 트리거는 문서, 코드, 설정, 로그 등 특정 영역에 한정되지 않는다.
- AI는 트리거 입력 시 해당 트리거에 정의된 절차를 따른다.
- 문서에 정의되지 않은 단어를 임의의 트리거로 해석하지 않는다.
- 트리거 실행 중 충돌, 해석 불가, 정보 부족이 있는 경우에만 사용자에게 다시 확인한다.

## 트리거 목록

### 동기화
#### 의미
프로젝트 기준 문서와 현재 코드, 설정, 관련 결과물의 차이를 검토하고 필요한 동기화 작업을 수행한다.

#### 동작
- AI는 관련 기준 문서를 전체적으로 확인한다.
- AI는 현재 코드, 문서, 설정 상태와 비교한다.
- 차이는 신규, 수정, 삭제로 구분한다.
- AI는 식별된 차이를 기준으로 필요한 동기화 작업을 수행한다.

#### 예외
- 기준 문서 간 충돌이 있는 경우
- 삭제인지 누락인지 판단할 수 없는 경우
- 문서만으로 결정을 내릴 수 없는 경우

## 확장 규칙
- 새로운 트리거는 이 문서에 추가하여 관리한다.
- 새로운 트리거는 `의미`, `동작`, `예외`를 포함해야 한다.
- 기존 트리거의 동작을 변경할 경우 이 문서를 먼저 수정한다.

## 트리거 해석 원칙
- 한국어를 포함한 원본 트리거 명칭은 엄격하게 해석해야 하며, 예를 들어 `동기화`는 예약된 트리거로 정확히 인식해야 한다.

### 도큐멘트
#### 의미
새로운 기능, 정책, API, 설계 문서의 초안을 작성하기 위한 예약 트리거

#### 동작
- 사용자가 `도큐멘트`라고 입력하면 AI는 문서 초안 작성 절차를 시작한다.
- AI는 먼저 작성 대상의 성격을 파악한다.
- AI는 문서 초안 작성 전에, 구현과 설계에 필요한 누락 정보를 사용자에게 질문한다.
- 질문은 해당 기능에 실제로 필요한 요구사항을 구체화하는 방향으로 진행한다.
- AI는 사용자의 답변을 바탕으로 문서 초안을 작성한다.
- 작성된 문서는 적절한 위치의 `project/doc` 이하에 정리한다.
- 문서 생성 또는 수정 이후에는 관련 로그 및 AI 정리 문서도 동기화 대상이 된다.
- AI는 `도큐멘트` 트리거로 문서를 생성하거나 수정한 경우, 관련 `project/.ai/context` 정리 문서도 같은 작업 흐름 안에서 반드시 생성하거나 갱신해야 한다.
- 관련 `project/.ai/context` 문서가 없는 경우에는 새로 생성해야 하며, 기존 문서가 있으면 변경된 원본 문서 기준으로 즉시 갱신해야 한다.
- 사용자 기능이 포함된 기능 설계 문서를 작성하거나 수정한 경우에는, 대응하는 프론트엔드 문서도 같은 작업 흐름 안에서 함께 작성하거나 갱신해야 한다.
- 프론트엔드가 현재 범위에 포함되지 않는 경우에는, 문서 초안 안에서 그 제외 범위를 명시해야 한다.
- 실제 구현 구성을 설명하는 문서가 존재하는 경우에는, 설계나 코드 변경으로 현재 구성 정보가 달라질 때 해당 문서도 같은 작업 흐름 안에서 함께 갱신해야 한다.
- 실제 테스트 체계를 설명하는 문서가 존재하는 경우에는, 설계나 코드 변경으로 테스트 구조나 검증 흐름이 달라질 때 해당 문서도 같은 작업 흐름 안에서 함께 갱신해야 한다.
- 루트 실행 도구(`Makefile` 등)가 존재하는 경우에는, 실행 방식이나 검증 명령이 달라질 때 해당 실행 도구도 같은 작업 흐름 안에서 함께 갱신해야 한다.
- GitHub Actions 같은 CI/CD 워크플로가 존재하는 경우에는, 테스트/빌드/실행 검증 흐름이 달라질 때 해당 워크플로도 같은 작업 흐름 안에서 함께 갱신해야 한다.
- 기존 문서 역시 예외가 아니며, 필요할 경우 AI가 이동, 분류, 재배치하여 전체 구조를 정리할 수 있다.

### 현재구성
#### 의미
AI가 실제로 선택하여 현재 저장소에 반영한 프로젝트 구성을 정리하거나 갱신하기 위한 예약 트리거

#### 동작
- AI는 `project/doc` 이하의 현재 구성 문서를 확인한다.
- AI는 현재 코드, 설정, Docker, 문서 상태를 기준으로 실제 구성을 정리한다.
- AI는 최소 기술 기준 문서와 실제 구현 구성 문서를 구분해서 다룬다.
- AI는 실제 구현 구성이 바뀌었으면 관련 구성 문서, 작업 로그, AI 정리 문서를 함께 갱신한다.
- AI는 루트 실행 도구(`Makefile` 등)가 존재하는 경우 현재 구성과 실행 방식에 맞게 함께 점검하고 갱신한다.
- AI는 CI/CD 워크플로가 존재하는 경우 현재 구성과 실행 방식에 맞게 함께 점검하고 갱신한다.

#### 예외
- 현재 구성과 기준 문서 간 충돌이 있어 어느 쪽을 우선해야 할지 판단할 수 없는 경우
- 구현이 부분 상태라 현재 구성으로 확정하기 어려운 경우

### 현재테스트
#### 의미
AI가 실제로 작성하고 사용 중인 테스트 체계와 검증 흐름을 정리하거나 갱신하기 위한 예약 트리거

#### 동작
- AI는 `project/doc` 이하의 현재 테스트 문서를 확인한다.
- AI는 현재 테스트 코드, 실행 명령, 검증 흐름, Docker 기반 확인 절차를 기준으로 실제 테스트 상태를 정리한다.
- AI는 테스트 코드나 검증 방식이 바뀌었으면 관련 테스트 문서, 작업 로그, AI 정리 문서를 함께 갱신한다.
- AI는 테스트 실행 진입점으로 사용되는 루트 실행 도구(`Makefile` 등)가 있으면 함께 점검하고 갱신한다.
- AI는 CI/CD 워크플로가 존재하는 경우 현재 테스트 흐름에 맞게 함께 점검하고 갱신한다.

#### 예외
- 테스트 코드가 아직 거의 없어 현재 테스트 체계로 정리하기 어려운 경우
- 실행 환경 차이로 테스트 여부를 현재 상태로 확정하기 어려운 경우

#### 요구사항 확인 원칙
- AI는 기능 구현에 필요한 핵심 조건이 비어 있으면 문서를 임의로 확정하지 않는다.
- AI는 누락된 항목이 있을 경우, 문서 작성 전에 먼저 사용자에게 확인한다.
- AI는 질문을 일반적으로 하지 않고, 작성 대상에 맞는 구체적인 항목을 질문해야 한다.

#### 예시: 로그인 기능
로그인 기능 문서를 작성하려는 경우 AI는 필요 시 아래와 같은 항목을 사용자에게 확인한다.

- 로그인 ID는 무엇을 사용하는지
  - 이메일
  - 아이디
  - 전화번호
  - 그 외 식별자
- ID의 글자 수 제한은 어떻게 되는지
- 허용 문자 규칙은 무엇인지
- 비밀번호 최소/최대 길이는 얼마인지
- 비밀번호에 필요한 조합 규칙은 무엇인지
  - 영문
  - 숫자
  - 특수문자
- 비밀번호 재시도 제한이 필요한지
- 계정 잠금 정책이 필요한지
- JWT 토큰을 사용할지, 세션을 사용할지
- 액세스 토큰/리프레시 토큰 구조가 필요한지
- 로그인 유지 기능이 필요한지
- 로그아웃 처리 방식은 무엇인지
- 비밀번호 찾기 또는 재설정 기능이 필요한지
- 이메일 인증 또는 2차 인증이 필요한지

#### 질문 원칙
- AI는 문서 품질에 직접 영향을 주는 항목만 질문한다.
- 이미 원본 문서에 정의된 항목은 다시 묻지 않는다.
- 원본 문서와 현재 사용자 답변을 합쳐도 결정할 수 없는 항목만 질문한다.
- 질문 수는 필요한 범위로 제한하되, 구현에 핵심적인 항목은 빠뜨리지 않는다.

#### 예외
- 사용자의 답변만으로 문서 목적이나 범위를 특정할 수 없는 경우
- 어떤 종류의 문서인지 판단할 수 없는 경우
- 기존 문서와 충돌 가능성이 높은 경우
- 핵심 요구사항이 비어 있어 임의 작성 시 왜곡 가능성이 큰 경우

위 경우에만 AI는 추가 확인 후 진행한다.

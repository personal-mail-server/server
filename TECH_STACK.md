# TECH_STACK.md

## 문서 목적
이 문서는 본 프로젝트의 기술 스택, 기술 선택 권한, 구현 원칙, 운영 기준을 정의한다.  
AI는 본 문서를 기준으로 기술을 선택하고 코드를 작성한다.

---

## 기본 원칙
- AI가 작성하는 모든 코드는 반드시 `project/code` 또는 `project/.github` 이하에서만 작성한다.
- AI는 지정된 경로 외부에 구현 코드를 작성하지 않는다.
- 구현, 구조 설계, 코드 컨벤션, 디렉토리 구성, 의존성 선택, 세부 기술 판단은 전적으로 AI가 담당한다.
- 사람은 기획 및 설계만 담당하며, 코드 구현에 직접 개입하지 않는다.
- AI는 명시적으로 요구되지 않은 범위까지 임의로 확장하지 않는다.
- AI는 기술 선택 시 반드시 보안성, 안정성, 유지보수성, 검증 가능성을 우선한다.
- AI는 검증되지 않았거나 운영 리스크가 큰 기술보다, 널리 검증되고 장기 유지보수가 가능한 기술을 우선 채택한다.

---

## 백엔드
- 백엔드 언어는 Go를 사용한다.
- 백엔드 프레임워크는 Echo Framework를 사용한다.
- API는 사용자가 직접 확인하고 테스트할 수 있도록 Swagger를 제공해야 한다.
- Swagger 생성 도구 및 구성 방식은 AI가 판단하여 결정한다.
- 백엔드 구조, 패키지 분리 방식, 계층 구조, 에러 처리 방식, 설정 관리 방식은 AI가 판단하여 결정한다.

---

## 프론트엔드
- 프론트엔드는 전적으로 AI가 판단하여 결정한다.
- 여기에는 언어, 프레임워크, 디렉토리 구조, 상태 관리 방식, 빌드 방식, 테스트 방식이 포함된다.
- UI/UX 설계와 구현 역시 전적으로 AI가 담당한다.
- 사용자는 UI/UX 전문 영역에 개입하지 않으며, AI는 사용성과 일관성을 고려하여 프론트엔드를 설계해야 한다.
- 프론트엔드는 프로젝트 목적에 맞아야 하며, 유지보수성, 보안성, 확장성을 저해해서는 안 된다.

## 백엔드-프론트엔드 연동 원칙
- API를 하나 구현할 경우, 백엔드 구현만으로 작업을 완료한 것으로 간주하지 않는다.
- 해당 API를 사용하는 프론트엔드도 함께 개발되어야 한다.
- AI는 백엔드와 프론트엔드가 실제로 연결된 상태를 기준으로 작업을 완료해야 한다.
- 사용자가 최종 확인할 때에는 구현된 기능이 웹에서 실제로 동작하는 형태여야 한다.
- Swagger는 API 확인 및 테스트 용도로 제공하되, 실제 사용자 기능은 프론트엔드에서도 확인 가능해야 한다.

## 실행 및 검증 원칙
- 모든 작업은 Docker를 통해 실행 및 확인 가능해야 한다.
- 사용자가 최종적으로 결과를 확인할 때에는 Docker 외의 추가 설치가 없어야 한다.
- 로컬 환경에는 가능한 한 영향을 주지 않아야 한다.
- Go, 프론트엔드 언어, 기타 런타임, 패키지 매니저, CLI 도구가 로컬에 설치되어 있지 않아도 실행 및 확인이 가능해야 한다.
- 프로젝트 실행, 테스트, 빌드, 확인에 필요한 의존성은 Docker 내부에서 해결되어야 한다.
- 로컬에 특정 라이브러리나 개발 도구가 없어도 최종 사용자의 확인에는 지장이 없어야 한다.
- 사용자의 최종 확인은 가능하면 단일 Docker 명령어 한 줄로 웹에서 확인 가능해야 한다.
- Docker를 실행하면 백엔드, 프론트엔드, 필요한 연결 구성이 함께 동작해야 한다.
- 사용자는 Docker 하나만 실행해도 전체 기능 흐름을 확인할 수 있어야 한다.
- AI는 로컬 의존형 구성보다 Docker 기반의 재현 가능하고 격리된 구성을 우선해야 한다.

## 데이터베이스
- 데이터베이스 종류는 AI가 판단하여 결정한다.
- 스키마 구조, 마이그레이션 방식, 인덱스 전략, 제약조건 설계, 데이터 무결성 보장 방식은 AI가 판단하여 결정한다.
- 데이터베이스 선택 시 반드시 안정성, 일관성, 운영 용이성, 장애 대응 가능성을 고려해야 한다.

## 인증 및 배포
- 인증 방식은 실제 관련 API를 작성하는 시점에 결정한다.
- 배포 환경 및 배포 방식은 실제 배포를 진행하는 시점에 결정한다.
- 따라서 AI는 인증과 배포에 대해 사전 확정된 것으로 간주하지 않는다.
- 단, 인증 및 배포 관련 구현이 필요한 시점에는 금융권 수준의 보안성과 통제 수준을 만족하는 방향으로 설계해야 한다.

## 테스트 원칙
- 테스트는 일부 주요 흐름만이 아니라, 가능한 모든 경우를 검증하는 것을 원칙으로 한다.
- 정상 케이스뿐 아니라 예외 케이스, 실패 케이스, 경계값, 잘못된 입력, 권한 문제, 상태 전이 문제를 모두 테스트 대상으로 본다.
- AI는 기능 구현 시 테스트 가능성을 반드시 함께 고려해야 한다.
- AI는 테스트 작성이 가능한 구조를 우선 설계해야 한다.
- 테스트 전략, 테스트 도구, 테스트 구조는 AI가 판단하여 결정한다.

## 운영 기준
- 이 프로젝트는 금융권에서 사용될 수 있을 정도로 엄격한 운영 기준을 전제로 한다.
- 따라서 AI는 모든 기술 선택과 구현에서 기능 편의성보다 보안성, 통제 가능성, 추적 가능성을 우선해야 한다.
- 개발 편의성만을 이유로 운영 리스크가 큰 선택을 해서는 안 된다.
- 임시 구현, 우회 구현, 추적 불가능한 구현, 책임 경계가 불명확한 구현은 지양한다.
- 장애 발생 시 원인 파악과 복구가 가능한 구조를 우선해야 한다.
- 변경 이력과 동작 의도를 설명 가능한 구조를 우선해야 한다.

## 보안 및 품질 기준
AI는 아래 기준을 항상 우선한다.

### 1. 보안성
- 최소 권한 원칙을 우선한다.
- 민감한 기능은 명시적 통제 지점을 가져야 한다.
- 입력값 검증, 권한 검증, 에러 처리, 데이터 보호를 기본 전제로 한다.

### 2. 안정성
- 장애 상황에서도 시스템 전체 영향이 최소화되도록 설계해야 한다.
- 실패 가능성을 전제로 예외 처리와 복구 가능성을 고려해야 한다.

### 3. 유지보수성
- 구조는 명확해야 하며, 책임 분리가 가능해야 한다.
- 코드와 설정은 추론 가능해야 하며, 변경 영향 범위를 파악할 수 있어야 한다.

### 4. 검증 가능성
- 모든 핵심 기능은 테스트 가능해야 한다.
- 동작이 복잡한 영역일수록 테스트 구조와 검증 포인트가 명확해야 한다.

### 5. 추적 가능성
- 장애, 실패, 상태 변화, 주요 처리 흐름은 추적 가능해야 한다.
- 운영 중 문제 발생 시 원인 분석이 가능하도록 구조를 설계해야 한다.

## 최종 원칙
- 백엔드는 Go + Echo Framework를 사용한다.
- Swagger는 반드시 제공한다.
- 프론트엔드의 기술 선택, UI/UX, 프로젝트 구조, 코드 컨벤션, 테스트 구조, CI/CD 등은 AI가 결정한다.
- API를 구현할 때에는 백엔드뿐 아니라 연결된 프론트엔드도 함께 개발되어야 한다.
- 인증과 배포는 필요한 시점에 결정한다.
- 테스트는 가능한 모든 경우를 검증하는 것을 원칙으로 한다.
- 모든 작업은 Docker로 실행 및 확인 가능해야 하며, Docker 외의 추가 설치를 요구해서는 안 된다.
- 사용자의 최종 확인은 가능하면 Docker 명령어 한 줄로 웹에서 확인 가능해야 한다.
- Docker 실행만으로 백엔드와 프론트엔드가 연결된 전체 기능 흐름을 확인할 수 있어야 한다.
- 모든 구현과 기술 선택은 금융권 수준의 엄격한 운영 기준을 만족해야 한다.
